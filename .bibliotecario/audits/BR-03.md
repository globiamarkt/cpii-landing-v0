#### Resumen ejecutivo
`tracking.js` centraliza la captura de parámetros de referencia (`ref`) desde la URL. Implementa una lógica de normalización (minúsculas y reemplazo de espacios por guiones bajos) y persiste el dato en dos niveles: `localStorage` (versión normalizada para lógica de negocio persistente) y `sessionStorage` (versión raw para el flujo de sesión actual). 
Este módulo resuelve la duplicidad detectada en la Sesión #1 y establece un patrón utilitario para el chasis.

#### Contexto de repo
Estado: activo — nuevo componente del chasis creado por Cursor en la Sesión 3.
Dependencias de entrada:
- Entorno global `window` (IIFE).
Dependencias de salida:
- Persistencia en `localStorage('cpii_ref')` y `sessionStorage('cpii_ref')`.
- Consumido por `index.html` y `simulator.html`.

#### Decisiones clave
- **Normalización Forzada:** El uso de `normalizeRef` asegura que los nombres de los referentes sean consistentes (ej. "Juan Perez" -> "juan_perez"), facilitando la integración con backend o analíticas futuras.
- **Dualidad de Almacenamiento:** El uso de `localStorage` asegura que el referente se mantenga si el usuario cierra el navegador, mientras que `sessionStorage` mantiene la captura original sin procesar para la sesión activa.
- **Silencio de Error:** Los bloques `try-catch` alrededor de las escrituras en almacenamiento previenen errores en entornos con cookies/storage bloqueados (Incógnito, etc.).

#### Arqueología de Marca
- El componente se registra bajo el namespace `global.CPIITracking`, siguiendo un patrón de "módulo de chasis" que evita la contaminación del scope global directo.

#### Potenciales conflictos y refactors sugeridos (para Agentes de dirección)
- **Extensibilidad:** Actualmente solo captura `ref`. Podría expandirse para capturar otros parámetros UTM o `campaign_id` de forma genérica.
- **Consistencia de Naming:** Mientras que i18n usa `cpii_lang`, este usa `cpii_ref`. Es consistente con el prefijo del proyecto.

#### Impacto en chasis
`tracking.js` actúa como el **sensor de entrada** del chasis. Se encarga de identificar el origen del tráfico e inyectar ese contexto en el estado persistente del navegador, permitiendo que el resto de archivos HTML personalicen el flujo basándose en el referente.
